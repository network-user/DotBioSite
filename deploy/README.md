# Деплой на свой сервер

Сайт - статика (Astro SSG). На сервере не крутится Node, отдаётся готовый
`dist/` лёгким веб-сервером **Caddy** с автоматическим HTTPS. Node нужен только
в момент сборки, после неё `node_modules` удаляется - на диске остаётся минимум.

## Что окажется на диске после деплоя

| Что                                | Примерно                            |
| ---------------------------------- | ----------------------------------- |
| Бинарь Caddy                       | ~40 МБ                              |
| Node 20 (для будущих пересборок)   | ~120 МБ                             |
| Исходники репозитория              | несколько МБ                        |
| Готовый `dist/` (то, что отдаётся) | несколько МБ                        |
| `node_modules`, npm/apt-кэш        | **0** - чистится сразу после сборки |

Пиковый расход диска - только во время `npm ci` (на время сборки появляется
`node_modules` ~300-500 МБ, потом удаляется).

## Предусловия

1. Сервер на **Ubuntu/Debian**, доступ по SSH с `sudo`/root.
2. **Домен уже указывает на сервер**: A-запись (и AAAA, если есть IPv6) ведёт на
   IP сервера. Проверить: `dig +short ТВОЙ_ДОМЕН` должен вернуть IP сервера.
3. Порты **80 и 443** открыты (firewall/группа безопасности). Они нужны Caddy
   для выпуска и продления TLS-сертификата Let's Encrypt.

## Деплой - 3 команды

```bash
# 1. Поставить git, если его нет
sudo apt-get update && sudo apt-get install -y git

# 2. Склонировать репозиторий (рекомендуемый путь - /srv, права под Caddy там корректные)
sudo git clone <URL_РЕПОЗИТОРИЯ> /srv/dotcore
cd /srv/dotcore

# 3. Запустить настройку, передав домен
sudo bash deploy/setup.sh example.com
```

Всё. Скрипт сам поставит Node и Caddy, создаст `.env`, соберёт сайт, почистит за
собой и поднимет Caddy с HTTPS. Через несколько секунд - `https://example.com`.

> Домен можно не передавать аргументом - тогда скрипт спросит его интерактивно.
> Первый запрос к сайту может секунду подождать, пока Caddy выпускает сертификат.

## Обновить сайт

После изменений в репозитории:

```bash
cd /srv/dotcore
sudo bash deploy/update.sh
```

Скрипт сделает `git pull`, пересоберёт статику, почистит `node_modules` и
перезагрузит Caddy. Node и Caddy не переустанавливаются.

## Настроить контент (имя, соцсети, фото, email)

По умолчанию сайт собирается на дефолтах - этого достаточно, чтобы он работал.
Чтобы появились ссылки на репозитории, соцсети, фото и кнопка email, отредактируй
`.env` в корне репозитория и пересобери:

```bash
cd /srv/dotcore
sudo nano .env          # заполни PUBLIC_GITHUB_USER, соцсети, AUTHOR_EMAIL и т.д.
sudo bash deploy/update.sh
```

Список и назначение переменных - в `AGENTS.md` (раздел «Переменные окружения»).
Поля `PUBLIC_AUTHOR_NAME_RU/EN` оставляй либо непустыми, либо вовсе убери из
`.env` (пустая строка уронит сборку - сработает дефолт `.ядро`/`.core`).

## Автодеплой из GitHub (CI)

`update.sh` можно дёргать не руками, а автоматически при каждом push в main -
GitHub Actions (`.github/workflows/deploy.yml`) сначала гоняет lint/type-check/
build как gate (без реального `.env` - просто проверка, что код валиден), а
затем по SSH выполняет на сервере тот же `sudo bash deploy/update.sh`, что
описан выше. Pull request'ы деплой не триггерят - только gate.

Настройка (один раз, после `deploy/setup.sh`):

```bash
cd /srv/dotcore
sudo bash deploy/ci-setup.sh
```

Скрипт создаёт отдельного пользователя `ci-deploy` с узкой sudo-привилегией -
ему разрешено запускать без пароля только `deploy/update.sh`, ничего больше.
В конце выводит, какой SSH-ключ сгенерировать и какие секреты добавить в
GitHub (`DEPLOY_SSH_KEY`, `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_REPO_PATH`).

Ручной запуск `sudo bash deploy/update.sh` по-прежнему работает и после
настройки CI - оба пути используют один и тот же скрипт.

## Ещё сильнее сэкономить диск (опционально)

Если обновляешь сайт редко, после деплоя можно снять и Node - освободит ~120 МБ.
Перед следующим `update.sh` его придётся поставить обратно (это сделает
`setup.sh`, либо вручную через NodeSource):

```bash
sudo apt-get remove -y nodejs && sudo apt-get autoremove -y && sudo apt-get clean
```

## Защита от DDoS / флуда

Сразу честно: одиночный небольшой сервер не выдержит мощный объёмный DDoS - для
этого нужен внешний CDN/скрабинг. Цель здесь скромнее - «не так просто положить»:
отсечь типовые флуды, slowloris и одиночные «долбящие» источники, не нагружая
диск и RAM. Защита идёт слоями.

### Слой 1 - таймауты Caddy (включён всегда)

Уже в `Caddyfile.tmpl`: ограничения на чтение заголовков/тела, запись и idle.
Это главный рычаг для одиночного сервера против slowloris (атака удержанием
множества «медленных» соединений). Плюс лимит размера тела запроса и заголовков.
Ничего делать не нужно - применяется при `setup.sh`.

### Слой 2 - rate-limit на уровне ядра (опционально, рекомендую)

```bash
cd /srv/dotcore
sudo bash deploy/harden.sh
```

Ставит отдельную таблицу nftables `ddos_guard`, которая ограничивает скорость
новых соединений на 80/443 **по каждому IP**. Один источник, бьющий быстрее
порога, отсекается; обычные посетители - нет (HTTP/2 переиспользует соединения).
Расход near-zero, правила переживают перезагрузку (systemd-юнит).

Скрипт **ничего не закрывает** (политика `accept`, дропает только флуд) и не
трогает SSH/ufw/docker - залочиться им нельзя. Тюнинг порога:

```bash
RATE=80 BURST=160 sudo bash deploy/harden.sh   # поднять лимиты
sudo systemctl disable --now ddos-guard.service # снять защиту
```

### Слой 3 - Cloudflare free (самая сильная защита, 0 нагрузки на сервер)

Единственный способ реально пережить объёмный DDoS на дешёвом сервере - спрятать
его за прокси Cloudflare (бесплатный тариф). Трафик идёт через сеть Cloudflare,
до сервера доходит уже отфильтрованный, IP сервера скрыт.

1. Добавь домен в Cloudflare, переключи NS на их.
2. A/AAAA-запись на IP сервера - с **оранжевым облаком** (Proxied).
3. SSL/TLS mode - **Full (strict)** (у тебя уже валидный TLS от Let's Encrypt).
4. (Опц.) Закрой сервер так, чтобы 80/443 принимали только с IP Cloudflare -
   тогда напрямую по IP сайт не достать. Список IP - на сайте Cloudflare.
5. (Опц.) Включи «Under Attack Mode» при активной атаке.

Слои совместимы и складываются: Cloudflare режет основную массу, `harden.sh` и
таймауты Caddy - последний рубеж, если кто-то нашёл прямой IP.

## Если что-то не так

- **Сертификат не выпускается / сайт только по HTTP.** Проверь, что домен реально
  указывает на сервер (`dig +short ТВОЙ_ДОМЕН`) и порты 80/443 открыты. Логи:
  `journalctl -u caddy -f`.
- **403 / Forbidden.** Caddy (пользователь `caddy`) не может прочитать `dist/`.
  Скрипт делает `chmod -R a+rX dist`, но если репозиторий лежит под `/root`
  (права `700`), Caddy не пройдёт по пути. Держи репозиторий в `/srv` или
  `/var/www`.
- **Сборка падает по нехватке диска.** Освободи место (`df -h /`), удали старый
  `node_modules`/кэш: `rm -rf node_modules .astro && npm cache clean --force`.
- **Проверить конфиг Caddy вручную:** `caddy validate --config /etc/caddy/Caddyfile`.
- **Перезапустить Caddy:** `sudo systemctl restart caddy`.

## Как это устроено

| Файл                    | Назначение                                                                                   |
| ----------------------- | -------------------------------------------------------------------------------------------- |
| `deploy/setup.sh`       | Первичная установка + первый деплой (идемпотентно)                                           |
| `deploy/update.sh`      | Пересборка и редеплой уже развёрнутого сайта                                                 |
| `deploy/ci-setup.sh`    | Доступ для GitHub Actions - пользователь `ci-deploy` + узкий sudo только на `update.sh`      |
| `deploy/harden.sh`      | Анти-флуд на уровне ядра (nftables per-IP rate limit)                                        |
| `deploy/Caddyfile.tmpl` | Шаблон конфига Caddy (security-заголовки, кэш из `public/_headers`, анти-slowloris таймауты) |

Security-заголовки и правила кэширования перенесены из `public/_headers`
(синтаксис Cloudflare Pages, который обычный веб-сервер игнорирует) в
`Caddyfile.tmpl`. Меняешь политику заголовков - правь шаблон и перезапусти
`setup.sh`.
